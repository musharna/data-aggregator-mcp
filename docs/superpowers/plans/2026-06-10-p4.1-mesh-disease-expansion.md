# P4.1 — MeSH disease query-expansion (true recall, server-side & ontology-grounded)

**Status:** PLANNED 2026-06-10. Target v0.26.0. Subagent-driven TDD (impl → spec review → quality review).
**Decision:** P4.1 "true semantic recall" via the proven `organism=`/`_expand_organism` pattern, extended to
disease/phenotype through **MeSH**. Chosen over UBERON (needs a new OLS client) and LLM-expansion (duplicates the
client's own LLM; ungrounded synonyms). Recall the keyword window can't reach, grounded in a real ontology — value
the calling client can't cheaply replicate.

## What "recall" means here (and what already exists — verified live)

- `embeddings.py` `rerank` only **re-orders the already-fetched keyword window** (`router.py:310`, anchored on raw
  `query`). Zero added recall. Untouched by this wave.
- The ONLY recall-adding mechanism today is `_expand_organism` (`router.py:123`): resolve `organism=` via NCBI
  Taxonomy → canonical + synonyms → AND an OR-group into the query. This wave adds the **exact same shape** for a
  `disease=` param backed by MeSH.

## Live-verified MeSH-over-E-utilities contract (probed 2026-06-10 — do NOT recall, re-probe in the live test)

Two round-trips, both via the EXISTING `_eutils` JSON helpers (no XML parsing — _simpler_ than `taxonomy.py`):

1. **Canonical-descriptor lookup** — field-restrict the term so NCBI's automatic term-mapping (ATM) collapses a lay
   name to its one canonical MeSH descriptor UID:
   - `esearch db=mesh term="breast cancer"[MeSH Terms] retmax=1` → `idlist == ["68001943"]` (one hit).
   - Contrast: a _plain_ `term=breast cancer` returns 63 hits whose **top id is `68064726` = "Triple Negative Breast
     Neoplasms"** (a narrower subtype) — WRONG. The `[MeSH Terms]` field restriction is load-bearing; without it the
     top relevance hit is not the canonical descriptor.
   - Use the existing `_eutils.esearch(client, "mesh", f'{name}[MeSH Terms]', retmax=1)` (JSON mode, already wired).
   - No mapping → `idlist == []` → return `None` (mirrors taxonomy's negative path).

2. **Descriptor detail** — `esummary db=mesh id=<uid> version=2.0 retmode=json` via existing `_eutils.esummary`:
   - `ds_meshui` → canonical MeSH UI, e.g. `"D001943"`.
   - `ds_meshterms` → list; **`[0]` is the canonical descriptor name**, the rest are entry-term synonyms.
     (Live for `D064726`: `ds_meshterms[0] == "Triple Negative Breast Neoplasms"`, rest = "Triple-Negative Breast
     Cancer", "ER-Negative PR-Negative HER2-Negative Breast Cancer", …)
   - `ds_recordtype` → `"descriptor"` for real descriptors. Accept ONLY `descriptor` (MeSH also has qualifiers/SCRs
     that are not valid query-expansion anchors); anything else → `None`.
   - **LIVE-TEST OBLIGATION:** I verified `"breast cancer"[MeSH Terms]` → UID `68001943` / `D001943`, but only
     verified `ds_meshterms[0]` text for `D064726`. The gated live test MUST assert the real `D001943` summary:
     `resolve_mesh(client, "breast cancer")` → `ui == "D001943"`, `canonical == "Breast Neoplasms"`, synonyms
     non-empty. If NCBI returns a different canonical surface form, fix the **assertion** (expectation drift, like the
     PDB-casing catch in v0.25.0), not the code.

## Build (TDD — write the test first for each unit)

### 1. `src/data_aggregator_mcp/mesh.py` (NEW — mirror `taxonomy.py`, JSON not XML)

```python
@dataclass(frozen=True)
class MeshInfo:
    ui: str             # MeSH descriptor UI, e.g. "D001943"
    canonical: str      # ds_meshterms[0]
    synonyms: tuple[str, ...]   # ds_meshterms[1:]

_NEG = object()
_CACHE = TTLCache(maxsize=4096, ttl=3600.0)   # same as taxonomy.py

async def resolve_mesh(client, name) -> MeshInfo | None:
    key = name.strip().lower()
    if not key: return None
    cached = _CACHE.get(key)
    if cached is not MISS: return None if cached is _NEG else cached
    _count, ids = await _eutils.esearch(client, "mesh", f"{name}[MeSH Terms]", retmax=1)
    if not ids:
        _CACHE.set(key, _NEG); return None
    docs = await _eutils.esummary(client, "mesh", [ids[0]])
    info = _parse_mesh(docs)          # None unless recordtype==descriptor & ds_meshterms non-empty
    _CACHE.set(key, info if info is not None else _NEG)
    return info
```

- `_parse_mesh(docs)`: `doc = docs[0]` (guard empty → None); require `doc.get("ds_recordtype") == "descriptor"`;
  `terms = doc.get("ds_meshterms") or []`; if not terms → None; `ui = doc.get("ds_meshui")` (guard falsy → None);
  return `MeshInfo(ui, terms[0], tuple(terms[1:]))`.
- HTTP failures **propagate** (NOT cached) — identical to `taxonomy.resolve_taxon` (caller surfaces in errors).
- Negative results cached (the `_NEG` sentinel), repeated disease in one request = one round-trip pair.

## Spec contracts the reviewers must enforce

- **Fail-loud, not fail-soft here:** `disease=` is a SEARCH-INPUT expansion, NOT a resolve enricher — a MeSH failure
  must surface in `errors["mesh"]` and leave the query un-expanded (exactly like `_expand_organism`'s
  `errors["taxonomy"]`). This is the OPPOSITE of the resolve-enricher §8 fail-soft rule (trust/cite/format) — do not
  conflate them.
- **`[MeSH Terms]` field restriction is load-bearing** (top plain-relevance hit ≠ canonical descriptor).
- **Composition order:** disease expands the organism-expanded query (both AND-groups present together).
- **Continuation never re-expands** (page consistency): `disease_expansion=None`, `disease` read from cursor.
