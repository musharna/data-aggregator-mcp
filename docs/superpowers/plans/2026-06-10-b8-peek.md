# B8 — `peek`: pre-download column profiling (`operate(op="peek")`)

**Status:** PLANNED 2026-06-10. Target v0.35.0. Subagent-driven TDD (impl → spec review → quality review).
**Decision:** add a new `op="peek"` mode to the EXISTING `operate` tool — a normalized, pre-download **column
profile** for a remote tabular file: per-column type, null-rate, distinct-count, min/max, and numeric quartiles, via a
single DuckDB `SUMMARIZE` over the already-hardened in-memory engine. From the v4 memo
[[data-aggregator-competitive-analysis-v4-2026-06-10]] (B8, Frontier #7): "DESCRIBE + LIMIT + null-rate via the existing
operate remote-read engine, normalized across formats. Turns operate into a discovery-time advantage a gateway can't
match." It is a NEW OP, not a new tool — the memo says "turns _operate_ into" a discovery advantage; `operate` is
already the pre-download inspection surface, so `peek` belongs there (lowest surface: one enum value + one engine fn).

## Grounding (reuses shipped code — verified live this session)

- `operate.run(client, resource_id, op, *, file, query, n, columns)` (`operate.py`) dispatches the 4 modes
  `OPERATE_MODES = ("schema", "preview", "head", "sql")`. It resolves the id, `_select_file`s the operable tabular file
  (`.parquet/.pq/.csv/.tsv`, truthy url), enforces `WALL_TIMEOUT_S=30`, `_cap_result_bytes` (`RESULT_BYTE_CAP=5e6`), and
  stamps `result["file"]`/`result["op"]`. **`head`/`sql` are size-gated** by `SOURCE_BYTE_CEILING=100_000_000` (they
  eager-load the whole file into RAM); `schema`/`preview` are NOT (footer/sniff range-reads only).
- `duckquery._connect(url, file)` (`duckquery.py`) is the HARDENED engine: `CREATE TABLE data AS SELECT * FROM
read_parquet|read_csv_auto(...)` (eager source read while local FS is still enabled), THEN
  `SET disabled_filesystems='LocalFileSystem'` + `SET lock_configuration=true`. After that point the in-memory `data`
  table is queryable but the local FS is sealed and config is locked. `run_head`/`run_sql` build on it; `_run` returns
  `{columns, rows, truncated}` with rows JSON-shaped via `dict(zip(names, r))`.
- **`SUMMARIZE data` (DuckDB 1.5.3, verified live) returns one row per column** with EXACTLY these fields:
  `column_name, column_type, min, max, approx_unique, avg, std, q25, q50, q75, count, null_percentage`. `count` is the
  total row count (same for every column); `null_percentage` is the null fraction (e.g. `33.33`); numeric-only fields
  (`avg/std/q25/q50/q75`) are `NULL` for VARCHAR columns. It runs on the in-memory `data` table → no FS access → works
  AFTER the lockdown (no new injection surface; `peek` takes NO user SQL).
- `tabular.schema(url, file)` already returns `{format, columns:[{name,type}], row_estimate}` from the Parquet
  footer / CSV sniff — `peek` is the SCAN-based, statistics-bearing complement (footer schema can't give null-rate).

## New engine function — `duckquery.run_peek`

```python
async def run_peek(url: str, file: str) -> dict:
    return await asyncio.to_thread(_peek, url, file)

def _peek(url: str, file: str) -> dict:
    con = _connect(url, file)            # REUSE the hardened lockdown engine
    try:
        rel = con.execute("SUMMARIZE data")
        names = [d[0] for d in rel.description]
        raw = [dict(zip(names, r, strict=False)) for r in rel.fetchall()]
        row_count = con.execute("SELECT COUNT(*) FROM data").fetchone()[0]
    finally:
        con.close()
    profile = [_normalize_summary_row(d) for d in raw]
    return {"row_count": int(row_count), "columns": profile}
```

- `_normalize_summary_row` maps each SUMMARIZE row to a JSON-safe profile entry. **Type discipline (JSON + honesty):**
  - `column_name` → str, `column_type` → str (the DuckDB type).
  - `null_percentage` → `float` (DuckDB hands back a Decimal/str — `float(x)`; keep the field name explicit, e.g.
    `null_percentage`).
  - `approx_unique` → `int | None` (it is an APPROXIMATE distinct count — name it `approx_unique` so we never imply
    exactness).
  - `min`/`max` → `str | None` (column-type-dependent values; stringify for a uniform wire type, `None` stays `None`).
  - numeric stats `avg`/`std`/`q25`/`q50`/`q75` → `str | None` (present for numeric columns, `None` for text — do NOT
    fabricate `0`; a `None` means "not applicable to this column type", surfaced honestly).
  - `count` (per-column total) is redundant with top-level `row_count`; OMIT it per-column to avoid implying it is a
    non-null count (honesty — `count` in SUMMARIZE is NOT the non-null count). The null _rate_ + `row_count` already
    give the user non-null counts if they want them.
- PURE-ish: same I/O profile as `run_head` (one remote eager read inside `_connect`, then in-memory). No user input
  beyond the file → nothing to sanitize.

## operate.py wiring

- `OPERATE_MODES = ("schema", "preview", "head", "sql", "peek")`.
- In the size-gate, add `peek` to the eager-load set: `if op in ("head", "sql", "peek"):` — `SUMMARIZE` scans the whole
  materialized table, so it has the SAME RAM profile as head/sql and MUST honor `SOURCE_BYTE_CEILING`. (schema/preview
  stay ungated.)
- In `_go()`, add `if op == "peek": return await duckquery.run_peek(url, target.name)`. `peek` ignores
  `query`/`n`/`columns` (document — it takes none). `_cap_result_bytes` is a no-op for peek (no `rows` key) but harmless;
  the per-column profile is bounded by column count, not row count, so no extra cap needed.

## server.py wiring

- Extend the `operate` `op` enum to `["schema", "preview", "head", "sql", "peek"]`.
- Extend the `operate` tool description with one clause: `op='peek' profiles every column WITHOUT downloading —
type, null-rate, approximate distinct count, min/max, and numeric quartiles (a DuckDB SUMMARIZE; like head/sql it
reads the whole file, so it honors the source-size ceiling).` Keep the existing op docs intact.
- No new tool, no model change (operate returns a raw `dict`, not a pydantic model — confirm: the `case "operate"`
  handler returns `await operate.run(...)` directly).

## Tests (tests/test_duckquery.py + tests/test_operate.py, mirror existing style)

- **Engine (`run_peek`) on a real local fixture** (write a tiny Parquet via pyarrow + a tiny CSV to a tmp path, address
  via `file://` — the existing operate/duckquery tests already do this; reuse their fixture helper):
  - returns `{"row_count": N, "columns": [...]}`; `row_count` equals the real row count.
  - per-column: a column with a known null fraction → `null_percentage` is the correct float (e.g. 1 null in 3 →
    ~33.33); a fully-populated column → `0.0`. **Assert the null-rate is computed, not guessed.**
  - `approx_unique` is an int (named approx — assert the KEY is `approx_unique`, documenting non-exactness); for a tiny
    fixture it equals the true distinct count.
  - a VARCHAR column → numeric stats `avg/std/q*` are `None` (not `0`/fabricated); a numeric column → they are present
    (str).
  - **per-column `count` is ABSENT** (we omit it to avoid the non-null-count misread).
  - Parquet and CSV fixtures yield the SAME profile shape (normalized across formats — assert key parity).
- **Lockdown intact:** after a `run_peek`, the engine is the same hardened `_connect` path — add/confirm a test that the
  FS stays sealed (peek itself takes no SQL, so assert the existing lockdown test still passes and that `run_peek` does
  not re-enable the FS). (Reuse the existing `disabled_filesystems` assertion pattern.)
- **operate.run integration:** `op="peek"` routes to the engine and stamps `result["op"]=="peek"` + `result["file"]`;
  an unknown-file / non-tabular file fails loud (reuse `_select_file` errors); `peek` IS size-gated — a source over
  `SOURCE_BYTE_CEILING` raises `OperateNotSupportedError` (monkeypatch `_source_size` like the head/sql ceiling test).
- **server wiring:** the `operate` inputSchema `op` enum includes `peek`; `OPERATE_MODES` and the enum agree (assert
  the tool enum == `operate.OPERATE_MODES` to prevent drift — or at least that `peek` is in both).
- **Real-execution check (gated `DATA_AGGREGATOR_MCP_LIVE=1`, mirror existing live operate tests):** `operate.run` with
  `op="peek"` against a REAL small remote Parquet (the same fixture id the existing live operate test uses) → a
  well-formed profile with a real null-rate and distinct count for ≥1 column. Catches a real remote-Parquet shape a
  local fixture hides.

## Explicitly OUT of scope (deferred)

- **Parquet-footer fast path** — null_count/min/max live in the Parquet footer (range-reads only, no full load), which
  would let `peek` skip the size gate for Parquet. Deferred: it splits Parquet vs CSV into two fidelity paths, breaking
  the "one normalized answer" property; SUMMARIZE-for-both is the honest, uniform v1. Note in the CHANGELOG as a future
  optimization.
- **Histogram / value-frequency / top-k** per column — richer profiling; not needed for the discovery use case.
- **Cross-file or cross-record profiling** — peek is single-file, like the rest of operate.

## Spec contracts reviewers must enforce

- **New OP, not new tool** — `peek` is one enum value + one engine fn; reuses `_select_file`, `_connect`,
  `WALL_TIMEOUT_S`, `SOURCE_BYTE_CEILING`. No duplicated id/file/extra machinery.
- **Size-gated** — `peek` eager-loads (SUMMARIZE scans), so it MUST honor `SOURCE_BYTE_CEILING`, exactly like head/sql.
- **Honesty in naming** — `approx_unique` (never "distinct"/"unique" bare); `null_percentage` a real computed float;
  numeric stats `None` for text columns (never fabricated `0`); per-column `count` OMITTED (SUMMARIZE `count` is total,
  not non-null — surfacing it as "count" would mislead).
- **Lockdown preserved** — `peek` runs `SUMMARIZE` on the in-memory `data` table AFTER the FS lock; it must not move or
  weaken the `disabled_filesystems`/`lock_configuration` sequence in `_connect`.
- **Normalized across formats** — Parquet and CSV yield the same profile keys; assert parity.
- **Additive** — the 4 existing ops are byte-identical; `peek` is purely new.
